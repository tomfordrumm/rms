# Phase 1: Создание ресторана и базовые настройки

## Суть задачи
Реализовать в админке полноценный функционал первичного заполнения и редактирования ресторана с базовыми настройками:
- name, description
- slug (для публичных страниц, с авто-генерацией и ручной правкой)
- logo и cover
- контакты (в удобном формате)
- часы работы по дням недели (минимум: `open_time` + `close_time`)
- перерывы/выходные (опционально)
- закрытия на конкретные даты (`closed_dates`)

Ключевой сценарий:
- отдельной страницы “Создание ресторана” нет;
- после регистрации пользователь сразу попадает на страницу “Настройки ресторана”;
- пока ресторан не создан (не заполнены обязательные поля), доступ к остальной системе заблокирован;
- после первичного сохранения та же страница используется для дальнейшего редактирования.

## Детальный план выполнения (чек-лист)

### 1. Проектирование данных и миграции
- [ ] Зафиксировать структуру хранения для ресторана:
  - [ ] `name` (required)
  - [ ] `description` (nullable)
  - [ ] `slug` (unique, indexed)
  - [ ] `logo_path` (nullable, отдельная колонка)
  - [ ] `cover_path` (nullable, отдельная колонка)
  - [ ] `contacts` (JSON/text)
  - [ ] `working_hours` (JSON по дням недели)
  - [ ] `breaks` (JSON, опционально)
  - [ ] `closed_dates` (JSON)
- [ ] Создать миграцию таблицы `restaurants`.
- [ ] Добавить индексы и ограничения (unique slug, FK, каскады при необходимости).

### 2. Модель и связи
- [ ] Создать модель `Restaurant`.
- [ ] Настроить `fillable/casts` (JSON-поля, даты, строки путей файлов).
- [ ] Описать связи в модели ресторана:
  - [ ] `user`
  - [ ] `tables`
  - [ ] `reservations`
  - [ ] `categories`
  - [ ] `dishes`
- [ ] Добавить аксессоры/мутаторы при необходимости (например, нормализация slug).

### 3. Backend: контроллер и действия
- [ ] Создать/доработать `RestaurantController`.
- [ ] Реализовать действия:
  - [ ] `edit/settings` (единая форма настроек)
  - [ ] `store` (первичное сохранение ресторана)
  - [ ] `update` (последующее редактирование настроек)
- [ ] Реализовать `upsert`-подход на одном экране (создание/редактирование в зависимости от наличия ресторана).
- [ ] Реализовать сохранение `logo` и `cover` в хранилище с возвратом путей.
- [ ] Добавить логику генерации slug из name (если slug не передан вручную).
- [ ] Добавить обработку конфликтов slug (суффиксы/ошибки валидации).

### 4. Валидация и правила
- [ ] Создать Form Request(ы) для `create/update`.
- [ ] Настроить обязательные правила:
  - [ ] `name` — `required|string|max:...`
  - [ ] `slug` — `nullable|string|alpha_dash|unique:restaurants,slug,...`
  - [ ] `open_time/close_time` — корректный формат времени
  - [ ] `logo/cover` — `image|mimes:jpeg,png,webp|max:...`
- [ ] Проверить бизнес-правила расписания:
  - [ ] `open_time < close_time` для рабочего дня
  - [ ] корректный формат `closed_dates`
  - [ ] отсутствие дубликатов дат закрытия
- [ ] Подготовить понятные сообщения об ошибках для UI.

### 6. Роутинг, доступы и интеграция
- [ ] Добавить маршруты для страницы настроек и сохранения (`store/update`).
- [ ] Ограничить доступ только админам/владельцам.
- [ ] Подключить страницу в навигацию админки.
- [ ] Проверить согласованность API-ответов с фронтом.
- [ ] После регистрации перенаправлять пользователя на страницу “Настройки ресторана”.
- [ ] Добавить middleware/guard “restaurant required”:

### 7. Тестирование
- [ ] Feature-тесты API:
  - [ ] успешное первичное создание ресторана на странице настроек
  - [ ] успешное обновление настроек

  - [ ] ошибки при некорректном времени
  - [ ] ошибки при неверном формате изображения/превышении размера
  - [ ] уникальность slug
  - [ ] сохранение `closed_dates`
- [ ] Тесты прав доступа (не-админ не может менять настройки).
- [ ] Тесты onboarding-потока:
  - [ ] после регистрации редирект на “Настройки ресторана”;
  - [ ] пользователь без ресторана не может открыть другие разделы;
  - [ ] после создания ресторана доступ к разделам открыт.
- [ ] При необходимости добавить unit-тесты для slug-генерации и нормализации расписания.

### 8. Definition of Done
- [ ] После регистрации пользователь принудительно проходит заполнение ресторана.
- [ ] До создания ресторана пользователь не может пользоваться остальными разделами системы.
- [ ] Ресторан создаётся и редактируется на одной странице “Настройки ресторана”.
- [ ] Валидация работает согласно требованиям.
- [ ] Логотип и обложка корректно загружаются и обновляются.
- [ ] Часы работы и даты закрытия сохраняются и читаются без ошибок.
- [ ] Все ключевые сценарии покрыты тестами и проходят локально.
